# veiland-sakura ‚Äî falling cherry-blossom petals plugin

A new reference plugin: textured sakura petals drifting *down* with tumble
and sway. Sibling to `veiland-particles` (which drifts dots *up*); separate
plugin per veiland's "one plugin = one job" convention, not a style flag on
particles.

## What it is

Each petal is a small textured quad sampling `assets/petal.png` (a soft pink
petal we generated procedurally ‚Äî `scripts/make_petal.py`, GPL/ours, no
external license). N petals fall from above the buffer to below it, each:

- **falling** (top ‚Üí bottom, opposite the dots' upward drift),
- **swaying** sideways (sin wobble, like wind),
- **tumbling** (per-petal rotation that advances over the fall, via the
  rotation-matrix pattern from `veiland-text`),
- **fading** in/out at the travel ends (reused from the dreamy-particles work),
- at a **per-petal cycle/size/phase**, randomised once at startup (the dots'
  PRNG + lifecycle, reused).

## Structure & reuse

`plugins/sakura/` ‚Äî new crate. Closest template is `plugins/particles`
(motion engine + buffer round-trip), plus:

- **Texture load** from `plugins/wallpaper`: `image` crate, PNG-only feature
  (`features = ["png"]`, the CVE-surface minimisation per CLAUDE.md), decode
  to RGBA8, `glGenTextures` + `glTexImage2D`, `LINEAR`/`CLAMP_TO_EDGE`.
- **Per-quad rotation** from `veiland-text`: a `vec2(cos,sin)` rotation of
  the local corner offsets around the petal centre, computed CPU-side per
  petal per frame (cheap; N is small).

The motion engine overlaps the dots plugin ~80% (PRNG, cycle/phase, fade,
the vertex-fill loop). **Do not pre-extract a shared crate** (CLAUDE.md:
"don't refactor opportunistically"). If the copy-paste genuinely hurts later,
factor a tiny `particle-motion` helper then ‚Äî not now.

## Asset

- `plugins/sakura/assets/petal.png` ‚Äî 96√ó96 RGBA, committed. Generated by
  `plugins/sakura/scripts/make_petal.py` (stdlib-only PNG writer; rerun to
  retune shape/colour). The plugin loads it from a path **relative to the
  binary's config**, defaulting to the bundled asset, configurable via
  `petal_path` (like wallpaper's `path`) so packagers can relocate it.
- Petal is authored already-pink; the shader can still tint via `color`
  (multiplied), default white = no tint.

## Rendering model

Per petal, per frame (in `update_vertices`):

1. `phase = ((now + t_offset) % cycle) / cycle`, **but falling**: y goes
   top‚Üíbottom. `cy = -r + phase * (h + 2r)` (phase 0 = just above the
   buffer, phase 1 = just below).
2. `cx = x_norm*w + drift*phase + WOBBLE*sin(phase*TAU + wobble_phase)` ‚Äî
   same sway as dots.
3. `rotation = rot0 + spin * phase` (radians); `spin` randomised per petal,
   signed (some tumble CW, some CCW). `(cos, sin)` computed once per petal.
4. Build the quad: local corners `(¬±r, ¬±r)`, **rotated** by `(cos,sin)`,
   then placed at `(cx, cy)`, converted to clip space. Quad carries petal
   UVs `(0..1)` for the texture, plus the fade alpha (per-vertex, like dots).
5. Vertex layout: `px, py, u, v, fade` = 5 floats/vertex, 6 verts/petal.

Shader:
- **VS**: passthrough clip-space `a_pos`, UV, fade (the rotation is baked
  into `a_pos` CPU-side, so the VS stays trivial ‚Äî no rotation uniform).
- **FS**: `texel = texture2D(petal, uv); a = texel.a * u_color.a * fade;`
  premultiplied out: `gl_FragColor = vec4(texel.rgb * u_color.rgb * a, a)`.
  (Premultiplied, matching the pipeline ‚Äî the core composites with
  `glBlendFunc(ONE, 1-SRC_ALPHA)`. See [[project_premultiplied_alpha]].)

## Config (TOML `[plugin.config]`)

- `count` (u32, default ~25) ‚Äî fewer than dots; petals are bigger.
- `petal_path` (string, default bundled `assets/petal.png`).
- `color` ([f32;4], default `[1,1,1,1]`) ‚Äî tint √ó the texture.
- `size_px` (f32, default ~22) ‚Äî petal half-extent in logical px, √óscale.
- `fall_seconds` min/max ‚Äî cycle range (e.g. 9‚Äì16s; petals fall slower and
  more languidly than dots rise). Constants for v1, like the dots' cycle.

## Z-order

In the Shinkai toml, sakura sits with the other atmosphere ‚Äî above wallpaper
+ vignette, around the particles' z (20‚Äì25). Decide whether petals go in
front of or behind the dot motes; likely just above particles so both read.

## Milestone fit

Reference-plugin territory (post-M11). Not on the original M11 list (that was
wallpaper/clock/particles/vignette) ‚Äî this is an *additional* reference
plugin in the same spirit. No protocol or host changes: sakura is a pure
consumer of existing Configure/FrameDone/Buffer flow, like every other
plugin.

## Commits (each leaves the tree runnable)

a. **Scaffold + asset**: new crate (`Cargo.toml`, `main.rs` skeleton that
   spawns, allocates a dmabuf, clears transparent, round-trips a buffer ‚Äî
   no petals yet), commit `make_petal.py` + `petal.png`. Validates the
   plugin loads and composites an empty buffer.
b. **Texture load + one static petal**: decode the PNG, upload, draw a
   single centered textured quad. Validates the asset path + sampling.
c. **Motion engine**: PRNG, per-petal Particle struct, falling + sway +
   fade, N petals. (No rotation yet.)
d. **Tumble**: per-petal rotation baked into the quad corners.
e. **Config surface**: count, size, color, petal_path, fall range.
f. **Shinkai toml**: add a `[[plugin]]` sakura block; tune count/size/z so
   it composes with the existing scene.

## Validation

Lock with the Shinkai config on both compositors (Hyprland + Sway). Petals
should fall, sway, and tumble independently, fading in/out at the edges,
reading clearly as cherry blossoms against the wallpaper. Tune count/size/
fall-speed to taste (all config). Watch the same ~10-13fps host pacing cap
the dots hit ‚Äî petals are bigger/slower so choppiness is less obvious.
